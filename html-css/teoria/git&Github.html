Git & GitHub

Git

O Git é um software de controle de versão e tem a finalidade de gerenciar diferentes versóes de um documento. Sendo assim, ele "salva" as diferentes versões de um projeto que está sendo manipulado ou em desenvolvimento. O git é um repositório local, que fica na maquina do desenvolvedor.

GitHub

O github é um serviço de web compartilhado para projetos que utilizam o git para versionamento. Trata-se de uma rede social de code, onde pode-se compartilhar e acessar projetos. Ele é um repositório remoto.
Nele podemos criar um número ilimitado de repositórios, hospedar códigos-fonte (em todas linguagens), com uma das suas ferramentas - o GitHub Pages- podemos "hospedar" sites básicos de forma gratuita e fora que permite colaborar com outros projetos e realizar forks para participar de projetos de terceiros.

Instalando e utilizando o Git

Para instalar o git na máquina, basta baixar o mesmo no seu site oficial e seguir com a instalação. Em seguida, vamos instalar também o VS Code, que vai ser o software no qual vamos manipular de forma ativa os projetos.
Após instalados vamos abrir o Git e/ou powershell para dar início as configurações iniciais e essenciais para criarmos nossos projetos.

  1. 1° No prompt de comando no git, vamos configurar nossa autoria, ou seja, vamos nos registrar como autores de todos os projetos que seram criados na máquina. A linha de comando para isso é: 
                        
                        git config --global user.name "nomeDoUsuário"
                        git config --global user.email emailUsuário@mail.com
                        git config --global core.editor vim
  2. 2° Após realizar as configurações globais, vamos criar uma pasta e nela criar um repositório local para dar um lugar a um projeto: 
                        
                        mkdir git- nome-projeto -> CRIA UMA PASTA
                        cd nome-pasta -> ENTRA NA PASTA
                        cd.. -> VOLTA A PASTA
                        dirgit- nome-projeto/ -> ENTRA NA PASTA 
                        git init -> INICIA UM REPOSITÓRIO local
                        ls-la -> MOSTRA O REPOSITÓRIO/LISTA DE ARQUIVOS
                        echo/vi nome-arquivo -> i (entra no modo insert) -> #Conteúdo do documento... -> TeclaESC (sai do modo insert) -> :w(: = inicia um comando; w = escreve e/ou salva) -> q = sai do modo insert.
  3. 3° Depois de iniciar o repositório, podemos checar o status dos arquivos no diretório, adicionar e commitar as nossos arquivos. 
                        
                        git status -> VERIFICA O ESTADO DOS ARQUIVOS/DIRETÓRIO
                        git add nome-arquivo.txt OU git add . -> para adicionar um arquivo ou diretório (staged area)
                        git commit -m "mensagem sucinta do que foi feito nesse commit"
  4. 4° Pronto! Realizamos um commit do projeto na nossa máquina local, entretanto precisamos conectar com a nossa conta no GitHub para subir o projeto para o repositório remoto. Para isso, com uma conta e chave SSH (passo a passo localizado na aba 'help' do GitHub) vamos realizar os seguintes comandos no terminal: 
                        
                        cd~/.ssh
                        ls -> AQUI COPIAMOS O id_rsa.pub
                        cat id_rsa.pub -> Copiar a chave SSH e adicionar no GitHub
                        git remote add origin emailgithub@git.com
                        git push origin nome-branch

Comandos do terminal

Através do git podemos realizar muitos tipos de comandos no terminal, um desses tipos de comando são os comandos de origem informativa:

  • ls-la -> MOSTRA O REPOSITÓRIO/ LISTA DOS ARQUIVOS
  • git log - -> EXIBE O HISTÓRICO
  • git log -- caminho-arquivo -> EXIBE O HISTÓRICO DE UM ARQUIVO ESPECÍFICO
  • git log --author nome
  • git shortlog -> EXIBE EM ORDEM ALFABÉTICA OS AUTORES, NÚMERO DE COMMITS E EM QUAIS ARQUIVOS FORAM MANUZEADOS
  • git log --graph -> MOSTRA AS BRANCHES DE FORMA GRÁFICA
  • git show n°hash do commit
  • git diff -> MOSTRA AS ALTERAÇÕES FEITAS (ANTES DE ADICIONAR/SALVA-LAS) - SERVE PARA DAR UMA REVISADA NO QUE FOI FEITO
  • git diff --name-only -> MOSTRA APENAS O NOME DO ARQUIVO QUE FOI MODIFICADO.
  • git commit -am "" -> ADICIONA TODOS OS ARQUIVOS MODIFICADOS + MENSAGEM
Outro tipo de comandos são os relacionados as branches. As branches são ramificações oriundas do código base, onde cada membro da equipe de desenvolvimento tem a sua e realiza suas alterações, e quando aprovadas para entrarem de forma oficial no projeto são 'fundidas' a branch principal. Segue alguns comandos relacionados as branches:
  • git clone url-repositório-remoto -> CLONA O REPOSITÓRIO
  • git checkout-b 'nome-branch' -> CRIA UMA NOCA BRANCH E JÁ FICA NELA
  • git branch-D nome-branch -> APAGA UMA BRANCH
  • git branch -> MOSTRA TODAS BRANCHES EXISTENTES
  • git checkout nome-branch -> NAVEGA PARA OUTRA BRANCH
  • git stash -> CRIA UM STASH (UM STASH É UMA BRANCH TEMPORÁRIA QUE CONTÉM ALTERAÇÕES QUE NÃO FORAM COMMITADAS)
Também há comandos para desfazer commits e outras alterações nos arquivos manuseados:
  • git reset HEAD nome-arquivo -> REMOSVE A EDIÇÃO DO ARQUIVO DEPOIS DE SALVO, PORÉM O CONTEÚDO AINDA ESTÁ DENTRO DO ARQUIVO PODENDO SER REMOVIDO COM O 'git checkout'
  • git rest --soft nome-arquivo OU n°da hash (anterior a que foi mandada errado) -> RETORNA A UM COMMIT ANTES DO ATUAL E RESETA AS MUDANÇAS
  • git reset --mixed nome-arquivo OU n°hash -> RETORNA O ARQUIVO AO ESTADO ANTERIOR AO COMMIT (MODIFIED STAGE)
  • git reset --hard nome-arquivo OU n°hash -> IGNORA O ÚLTIMO COMMIT E RESETA A PENULTIMA VERSÃO DO CÓDIGO
  • git revert n°hash -> REVERTE O COMMIT FEITO NO N° DA HASH EM QUESTÃO
Ainda temos os comandos de união entre branches, que são dividodos em dois tipos, o merge e o rebase.
O Merge unifica as branches e suas linhas de desenvolvimento independentes em uma única ramificação. O merge adiciona um novo commit, garantindo que nenhuma alteração seja perdida.
Já o rebase unifica s branches criando um histórico linear, movendo a ramificação de funções para a ponta da ramificação master. O rebase comprime todas as alterações em um único "patch" e o integra na branch de destino. Ou seja, é uma forma mais arriscada e deve ser usado com cuidado, pois, ele altera o histórico de commits, reescrevendo todo o histórico dos commits.
  • git merge nome-branch -> UNI AS BRANCHES COM MERGE
  • git rebase nome-arquivo -> UNI AS BRANCHES COM REBASE
Por fim, temos outros comandos com finalidade de interagir com o repositório remoto:
  • git remote add origin endereço-SSH
  • git push -u origin nome-branch -> UPA OS COMMITS DO REPOSITÓRIO LOCAL PARA O REPOSITÓRIO REMOTO
  • git pull -> ATUALIZA O REPOSITÓRIO LOCAL COM AS ALTERAÇÕES FEITAS NO REPOSITÓRIO REMOTO
  • git cat nome-arquivo -> MOSTRA O CONTEÚDO DO ARQUIVO DO REPOSITÓRIO REMOTO
  • git fench origin branch -> PEGA TODOS OS DADOS DO REPOSITÓRIO REMOTO QUE VOCÊ NÃO AINDA NÃO TEM

Ciclo de vida do status dos arquivos

UNTRACKED

Trata-se do momento no qual o arquivo foi adiciona no repositório, mas ainda não foi 'visto pelo git', ou seja, o git não o reconhece dentro do seu repositório nenhuma versão do arquivo.

Unmodified

Esse estágio ocorre após adicionar o novo arquivo (git add/ add .) no git sem registrar nenhuma modificação após adicionado.

Modified

Quando após adicionado/commitado o arquivo passou por alguma modificação

Staged

Quando o arquivo modificado é recebido e criado uma versão dele. Após o arquivo receber um commit ele volta para o estágio unmodified (ou seja, não foi modificado desde sua última versão comitada).

Fork, Branches e Git Ignore

Fork

É uma funcionalidadee que copia umprojeto de terceiros, para que se possa trabalhar também no mesmo projeto. Quando finalizado, pode-se realizar um pull-request e contribuir com o projeto de alguém.

Já as branches como já abordadas anteriormente, são bifurcações do código-fonte que criam uma linha indiferente de desenvolvimento. Funciona como um controle de origem que permite separar a evolução do código principal de forma livre, sem afetar o trabalho do outro. Esse recurso viailiza testar novos recursos, e corrigir eventuais erros sem afetar o código-fonte do projeto.

Por fim, o Git Ignore é um arquivo oculto que indica quais arquivos ou tipos de arquivos não devem ser rastrerados para versionamento. Para isso basta salvar o arquivo a ser ignorado dentro do '.ignore' que também funciona com extensões. geralmente é utilizado para arquivos sensíveis (senhas) ou irrelevantes (node_module).
